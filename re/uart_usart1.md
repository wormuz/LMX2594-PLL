## UART Reverse-Engineering: Complete Findings

**Peripheral identity correction**: task brief said "USART3 @0x40004800" and "USART1 @0x40013800" — verified only 0x40004800 is actually wired up in code. It is **USART3** (STM32F103 map: USART1=0x40013800, USART2=0x40004400, USART3=0x40004800). No code anywhere in the 64KB image references 0x40013800 or 0x40004400 — **USART1 and USART2 are never initialized or touched**. The task brief's "USART1 @0x40013800" reference does not correspond to any real peripheral access; disregard it.

**1. Init — `FUN_08006a6c(uint baud)`, called from `FUN_08006410` (main loop) at 0x0800642a with `baud=0x2580` (9600 decimal) and from `FUN_08006364`'s sibling path via 0x0800640c:**
- Clocks: `FUN_08004564(0x40000,1)` → RCC_APB1ENR |= bit18 (USART3EN); `FUN_08004584(8,1)` → RCC_APB2ENR |= bit3 (IOPBEN, GPIOB clock)
- GPIO: two `GPIO_Init`-equivalent calls (`FUN_080023b2` = StdPeriph `GPIO_Init`) on GPIOB (base 0x40010C00):
  - Pin=0x400 (**PB10**), Speed=3 (50MHz), Mode=0x18 (`GPIO_Mode_AF_PP`) → **TX**
  - Pin=0x800 (**PB11**), Mode=4 (`GPIO_Mode_IN_FLOATING`) → **RX**
- `FUN_08003d08` = `GPIO_PinRemapConfig`-style AFIO write (value 0x27,1,1,1 — no remap bit set, default USART3 TX/RX = PB10/PB11, not remapped)
- `FUN_0800570c` = StdPeriph `USART_Init`: WordLength=8, StopBits=1, Parity=none (local_18/16/14/10=0), BaudRate param = 9600, Mode = 0xC (RX+TX enabled)
- `FUN_080056c0` = `USART_ITConfig(USART3, 0x525, ENABLE)` — enables an interrupt bit combo (0x525 encodes CR1 offset+bit5=RXNE-adjacent per the driver's `(reg>>5)` scheme → **RXNE interrupt enabled**)
- `FUN_080055a0` = `USART_Cmd(USART3, ENABLE)`

**PB11 conflict**: PB11 is simultaneously the bit-bang SPI **DATA** line to the LMX2594 (per verified HW facts) and now also USART3_RX. This is a real pin-sharing bug in this alpha firmware — every LMX SPI write will corrupt/be corrupted relative to whatever is on USART3_RX, and vice versa, unless one function is entirely unused at runtime.

**2. TX-only or RX+parser?** Both directions exist in code: a TX primitive at 0x080063f0 (polls USART3 SR bit6=TC or TXE, mask 0x40, then writes DR) used throughout the CLI for responses, and USART3 IRQ is enabled for RXNE (0x525 ITConfig). However — **the RXNE interrupt handler was never located**: `USART3_IRQHandler` vector slot in the vector table (checked 0x08000000-0x080000FF) contains only default/repeated stub pointers (0x080208DB pattern, a shared dummy handler), not a distinct USART3 ISR. No BL/call site or literal-pool reference to the byte-ingestion function `FUN_08005b58` (the CR/LF framer that fills the shared command buffer) exists anywhere in the 64KB flash image — exhaustively verified via hex search for its two possible odd/even Thumb addresses and via enumerating every one of the 3 total references to flag-word 0x20001DD8 and both references to buffer 0x2000288C. All of those references belong to: the parser consumer (`FUN_08006410`), the framer itself (`FUN_08005b58`), and the USB-CDC `SetLineCoding` handler (`FUN_080055b8`, which zeroes the flag as a side effect, not as producer).

**Conclusion — bug/gap, stated plainly, not softened**: USART3 RX is fully configured (GPIO, clock, baud, RXNE interrupt enable, USART_Cmd enable) but **I did not find the code that actually receives bytes into the command buffer** — neither a distinct IRQ handler nor a polling call site for USART3 SR/DR exists in the disassembled ranges I covered. The only confirmed producer path into the shared 200-byte command buffer at 0x2000288C / flag 0x20001DD8 is `FUN_08005b58`, and that function itself has zero callers in flash. This means either (a) it's genuinely dead code / never wired to USART3's ISR — the RXNE IRQ fires into the shared default handler and RX bytes are silently dropped, so USART3 is TX-only in practice despite RX being "enabled" at the register level, or (b) the real caller is reached through a computed/indirect call I have not located (e.g., a RAM-resident vector table entry not present in the static flash image, which would require live/runtime inspection to confirm). I have not proven which. This is a genuine unresolved gap, not "cosmetic."

**3. Command parser identity**: The parser in `FUN_08006410` (`start`, `stop`, `w1`, `w2`, `w3`, `wt`, `wv`) is **the same single parser and same shared RAM buffer** (0x2000288C, flag 0x20001DD8) for both channels — there is only one command buffer in the whole binary. USB-CDC and USART3 would feed the identical parser and command set if both were wired to the same framer; no second/different parser exists anywhere in the image.

**4. RX buffer/framing**: Buffer = 200 bytes (0xC8) at RAM 0x2000288C, flat byte array, no length-prefix framing. Terminator = **CR then LF** (`\r\n`): first `\r` sets a "seen-CR" state bit (0x4000); the following byte must be `\n` to set the "command ready" bit (0x8000) that `FUN_08006410` polls; any other byte following `\r` resets state to 0 (frame abort). Index field is the low 14 bits of the same 16-bit flag word, wrapping at 200.

Key addresses: USART3 init `0x08006a6c` (baud call sites `0x0800642a`, `0x08006426`=`mov r0,#0x2580`); GPIO/USART pool `0x08006b08`(GPIOB)/`0x08006b0c`(USART3); TX primitive `0x080063f0`-`0x0800640a`; framer `0x08005b58`-`0x08005bce` (flag ptr `0x08005bd0`→RAM 0x20001DD8, buf ptr `0x08005bd4`→RAM 0x2000288C); CLI parser `0x08006410`; CDC SetLineCoding `0x080055b8`.
